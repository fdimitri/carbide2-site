The [last post](/blog/forcing-https-traefik-redirect) ended on a victory lap:
HTTPS working, pages loading. We moved on.

Then the IDE wouldn't connect.

The dashboard rendered fine. The workspace WebSocket — terminal, filesystem,
agent, call signalling — sat there retrying forever. No error worth the name.
Just a socket that opened and instantly died.

## The cert we waved through was the whole problem

> A browser will let you **click through** an untrusted TLS cert for a page.
> It will **not** let you click through it for a `wss://` WebSocket. There is
> no bypass dialog. The handshake just fails.

`tls: {}` in Traefik means "terminate with whatever cert you have." Traefik's
fallback is a built-in self-signed cert for a hostname that isn't yours. For a
*page*, Chrome shows the interstitial and remembers your click-through for the
tab. WebSocket connections get strict validation with no UI and no override —
so `wss://` dies silently while the page loads fine. `openssl s_client` against
the live port confirmed it before we changed a line.

The fix: a locally-trusted cert (we used [mkcert](https://github.com/FiloSottile/mkcert))
installed as Traefik's **default certificate** via a `TLSStore` named `default`
in Traefik's namespace. One object covers every `tls: {}` route in the cluster —
dashboard and all workspaces — without touching each route individually.

## The actual crux: three machines, one trust store assumption

Development happens in WSL2. The cluster runs on **dev1**, a separate box on the
LAN. The browser is on the Windows host. None of them share a trust store.

We generated the cert on dev1 because that's where the cluster lives. dev1 got
the trusted CA. The Windows browser never saw it. The symptom looked like a cert
problem; the real shape was a *topology* problem. Deploying to a different
machine than you're developing on hides the CA installation behind a machine
boundary you don't discover until the browser refuses to connect.

Fix: copy the CA to the Windows host and import it into `Cert:\CurrentUser\Root`.
Terminal tools on dev1 point at the CA file directly. Restart the browser and the
socket connects.

## Two papercuts from switching hostnames

Moving from `localhost` to `dev1.frankd.local` found two more issues.

**Wrong cert SANs.** The first cert was auto-detected from `hostname -f`, which
returned bare `dev1` — not the FQDN the browser uses. A cert valid for the wrong
name is as useless as no cert. The deploy now takes a `PUBLIC_HOST` and puts it
in the cert's SANs.

**Rails 8 host authorization.** The app greeted the new hostname with
`Blocked hosts: dev1.frankd.local`. Rather than hand-editing an env var, the
operator now parses the host out of the public URL it already knows about and
appends it to each workspace pod's allowlist. Set `PUBLIC_HOST` once and the
cert, redirects, and host authorization all agree.

## One more thing

The cert step has ordering dependencies that loosely-coupled shell scripts
couldn't guarantee. So there's now a single `deploy.rb` that runs the whole
pipeline as an explicit, idempotent sequence. Yes, it's another entry point in
a graveyard of deploy mechanisms. A deploy you can run twice and get the same
answer is worth retiring a few scripts for.

## What we'd tell past us

- **"Click through" is a page-only escape hatch.** WebSockets get strict cert
  validation with no bypass.
- **`tls: {}` means "use the default cert" — so define the default.** A
  `TLSStore` named `default` covers every route at once.
- **The CA must be trusted where the browser runs.** In a WSL2 + remote-cluster
  setup, that's a third machine nobody thought to check.
- **Pick one name for the box and thread it everywhere** — cert SANs, public URL,
  and host authorization must all agree.
