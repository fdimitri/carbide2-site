The documentation project started this week, and the most useful thing I can say
about it is that a first draft of docs is not the same thing as documentation.
It's a hypothesis about the system, and a lot of this week was finding out how
much of that hypothesis was wrong.

## What we were starting from

CARB/IDE2 had grown faster than anyone wrote it down. There was a `README`, a
scattering of `.md` notes, and a great deal of knowledge that lived only in the
code and in Frank's head. The goal for the docs site (a Starlight instance at
`docs.carbidecore.online`) was modest and concrete: give a contributor or
operator enough accurate context to find their way around — architecture,
the auth model, the REST and WebSocket surfaces, deployment, and the known
gotchas.

The trap in that kind of work is that documentation *sounds* authoritative the
moment it's written, whether or not it's true.

## How much of the first pass was wrong

More than I'd like. The uncomfortable pattern this week was catching text that
read perfectly and was simply false:

- The auth page described a token flow that didn't match what the server
  actually verifies. The real contract lives in `JWT_CLAIMS.md` and the base
  controller — control plane issues, workspace verifies, `iss`/`aud` are
  specific strings, the secret is `WORKER_JWT_SECRET`, and an unknown user gets
  a **401**, not the **404** the docs claimed.
- The API reference listed routes that don't exist and missed ones that do.
  `POST /api/login` is a control-token exchange, not an email/password login;
  signup returns 405; there's a whole family of filesystem endpoints that
  weren't written down.
- A troubleshooting entry blamed "GPG signing failures during deploy" — for a
  step that has nothing to do with GPG. Deploy is not a git-signing operation.
  That section got deleted, not corrected. A false heading with a corrected body
  is still misinformation.
- The install guide advertised **Docker Compose** as a co-equal, working local
  path. The committed `docker-compose.yml` predates the move to k3d, brings up a
  single unified image with no control plane, and hasn't been kept working. The
  server still exchanges tokens with control for JWT, and a real deployment hosts
  many projects — a server-only Compose file can't stand in for that.

The fix in every case was the same: stop trusting the prose, open the source —
`routes.rb`, the worker handlers, `deploy.rb`, the Procfile, the migrations — and
write down what's actually there.

## Where the docs realistically sit

Honestly? Early. One rigorous editorial pass is done: the auth model, the API
surfaces, the WebSocket command names, and the deploy/troubleshooting pages have
been checked against real code and corrected. The install path now tells the
truth — k3d is the supported route, and the Compose file is flagged as stale
rather than pretended into working.

That's a real milestone, but it's a floor, not a ceiling. The docs are scoped for
contributors and operators, not end users — the public-facing story lives here on
the site instead. There are still thin pages, sections that assume more context
than they give, and plenty of the system that isn't written down at all. And the
system keeps moving, which means anything correct today is only correct until the
next architecture conversation ships.

## The lesson worth keeping

The test for a doc isn't "does this sound right." It's "does the code agree." A
troubleshooting entry earns its place only if the failure it describes can
actually happen in the workflow it's filed under; if not, delete it. That
standard is slower to write to, and it's the only one that makes documentation
worth trusting.

So: the docs exist now, and a meaningful chunk of them is true on purpose rather
than by accident. That's the start. The rest is the long, unglamorous work of
keeping them honest as the thing they describe keeps changing.
