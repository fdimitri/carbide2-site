Every workspace shell in CARB/IDE2 runs as a non-root user — `carbide`, uid 1000.
That's deliberate: the threat model is co-tenancy *inside* a workspace, so the
interactive terminal should never be root. The pod's project directory is mounted
at `/workspace`, and the user works there.

Except they couldn't. `touch` failed. `git commit` failed. The directory was
owned by `root:root`, and uid 1000 was locked out of its own files.

## Fetch the data before guessing

The tempting move is to theorize about the cascade of Kubernetes security
contexts and start editing YAML. Instead: `kubectl exec` into the live pod and
just *look*.

```
$ id
uid=1000(carbide) gid=1000(carbide) groups=1000(carbide)
$ ls -ld /workspace
drwxr-xr-x 5 0 0 4096 /workspace
$ touch /workspace/.wtest
touch: cannot touch '/workspace/.wtest': Permission denied
```

The pod spec already set `fsGroup: 1000`, which is *supposed* to make kubelet
chown the volume to gid 1000 at mount time. It hadn't. The directory's group was
still `0`.

The reason is in the storage class. k3d ships the `rancher.io/local-path`
provisioner, which creates the backing directory on the node as `root:root 0755`.
And kubelet's `fsGroup` recursive chown does **not** reliably apply to **subPath**
mounts — and `/workspace` is mounted via `subPath`. So the directory lands
root-owned, `fsGroup` notwithstanding, and the unprivileged user can't write.

## The half fix

The smallest change that makes the symptom go away is a root **initContainer**
that chowns the mount before the unprivileged shell starts:

```ruby
initContainers: pvc_mounts.empty? ? [] : [{
  name: 'fix-workspace-perms',
  image: SHELL_IMAGE,
  command: ['sh', '-c', 'chown 1000:1000 /workspace && chmod 0775 /workspace'],
  securityContext: { runAsUser: 0, runAsGroup: 0 },
  volumeMounts: pvc_mounts,
}]
```

It runs as root, fixes ownership, then the main container drops to uid 1000 onto
an already-writable directory. Re-exec into the pod:

```
$ ls -ld /workspace
drwxrwxr-x 5 carbide carbide 4096 /workspace
```

The directory is fixed. Victory?

## No — the contents were still root

```
$ ls -al /workspace
drwxrwxr-x 5 carbide carbide  4096 .
-rw-r--r-- 1 root    root     3175 README.md
-rw-r--r-- 1 root    root      635 astro.config.mjs
drwxr-xr-x 7 root    root     4096 src
```

The *directory* was now `carbide`-owned, but every file **inside** it was still
`root:root`. The initContainer fixed the mount point, not its contents — and the
user still couldn't edit a single imported file.

That's the difference between a symptom and a root cause. The mount-point chown
was real and worth keeping, but it papered over the actual bug: **the files were
being written as root in the first place.**

The chain runs like this. When a project is imported, the worker process — which
runs as root — records each file's POSIX ownership into the in-database
filesystem. Our [DB-backed filesystem](/blog/in-database-filesystem) then flushes
that content to disk, and the flusher faithfully re-applies the stored owner: uid
0. So the importer stamps root, the flusher honors root, and the file lands
unwritable by the user who's supposed to own it. The dir-level chown can't undo
that; it never touches the files.

## Where it stands

The mount-point fix is committed — it's correct and necessary on its own. The
deeper issue — imported files inheriting the importer's uid instead of the
workspace runtime user's — is filed as a tracked bug with a clear acceptance
test: a freshly imported project should be editable and committable from the
terminal without `sudo`. The fix belongs at import time (normalize ownership to
the runtime user) and/or in the flusher (map uid 0 to the runtime user), with one
source of truth for "who owns workspace files." The answer is almost certainly:
the person sitting at the terminal.

## The lesson, again

Two things made today go faster than it could have. First, reading the live
state before forming a hypothesis — one `kubectl exec` told us more than an hour
of reasoning about `fsGroup` semantics would have. Second, not stopping at the
first green checkmark. `ls -ld` looked fixed. `ls -al` told the truth. A fix that
addresses the symptom is worth shipping; mistaking it for the root cause is how
the bug comes back next week.
