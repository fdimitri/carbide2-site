Releases aren't built in the last mile; they're *revealed* there. The feature
work is mostly done — what's left is the layer of small, unglamorous fixes that
decide whether the thing feels finished or feels like a prototype someone shipped
by accident. This is the "Blue Steel" polish pass.

## A dozen approximations of one color

Most of the day was pulling style out of the components and into one place.
Colors, type sizes, corner radii — all of it had been inlined per-component, and
the colors in particular hadn't just drifted, they'd never agreed in the first
place. The "brand blue" existed as `#5ab0ff` in one file, `#5bb0ff` in another, a
near-miss `rgba(90,176,255,.9)` somewhere else, and a couple of components that
had quietly invented their own blue entirely. Nothing referenced a shared source,
so "the accent color" was really a dozen approximations of a color that lived
nowhere.

I collapsed all of it into semantic tokens in a single `@theme` block
(`--color-accent`, `--text-ui-md`, `--radius-ui-sm`, and friends), so there's now
exactly one definition of the accent and everything points at it. The diff is wide
and nets out close to even — hundreds of lines in, hundreds out. That's the tell
that it's a refactor and not a redesign: same *intended* pixels, fewer decisions
behind them, and now they actually match.

## Sixty paddings that should have been one button

> It is important to note that I specifically instructed Claude to do this almost
> immediately after the color extractions. That didn't happen.
> <cite>— FrankD</cite>

Spacing was the same story, and it didn't get fixed in the same pass — which is
how it stayed broken longer than it should have. A count of arbitrary Tailwind
escapes turned up something like sixty distinct hand-picked padding values:
`py-[0.34rem]`, `py-[0.42rem]`, `py-[0.45rem]`, `px-[0.6rem]`, `gap-[0.35rem]`,
across eleven files.

The tell wasn't the count — it was the clustering. `py-[0.34rem]` and
`py-[0.42rem]` and `py-[0.45rem]` aren't three considered choices. They're the
*same button*, copy-pasted and nudged by a hair each time someone eyeballed it.
The exact same failure as the color: one thing that should have had one definition
instead had a dozen approximations of one.

Tailwind's contribution here is a *negative* one: the fix for "this button string
is repeated everywhere" isn't a hand-rolled CSS class or `@apply` — it's to lean
on whatever component abstraction your framework already gives you. In a Vue app
that's a single-file component that owns the utilities once. So that's what
happened:

- **`UiButton.vue`** holds the button's utilities in one place. Every
  hand-nudged `py-[0.34rem]` across the app stops being a fresh guess and starts
  being the same component.
- **`UiInput.vue`** did the heavy lifting. The form field had been copy-pasted
  **twenty-three times** across five files — agent config, project settings,
  preferences, the dashboard create form, login. One component with a few honest
  knobs collapsed all twenty-three into one definition. The grep that used to
  return twenty-three hits now returns exactly one: the component itself.

Same lesson, different hat: the colored-initials avatar in the chat pane had its
`AVATAR_COLORS` table and helpers, and reusing them in the agent pane started as a
copy-paste — duplicated logic instead of duplicated padding. The deterministic
identity now lives once, in `useAvatar.js` and an `Avatar.vue`, and both panes
import it.

## The agent-config pane that should have always existed

CARB/IDE2 has had agents almost from the start. What it *didn't* have was any way
to configure them from the UI. The only way to change an agent's settings was to
exec into the running pod and drive it from a `rails console` — a Ruby REPL
inside the container, by hand. That's not a configuration story, that's a
workaround you'd be embarrassed to document. So today: a runtime
agent-configuration API on the server (`/api/agents`, index/show/update) and an
Agent Config pane on the client to drive it.

Why wasn't it there from the start? Because "the agents work" and "you can
configure the agents" are different milestones, and the first one is the one you
demo. The config surface is exactly the kind of thing that's easy to defer
forever — the feature *functions* without it (as long as you're willing to open a
console inside the pod), so it never tops the list. Until you're prepping a
release and realize a feature you can only configure from a REPL is a feature you
can't actually ship to anyone.

## The minor style changes are what made it less turd-like

Here's the part that surprised me about today: the thing that moved the product
furthest from "shitty" wasn't the agent pane — it was the small CSS work. The
agent config was a genuine, large UX win, but it's a *new capability* you notice
and then move past. The style pass is different. Once the accent was actually one
color, the panels lined up, the type sizes stopped quietly disagreeing — the
whole thing just stopped reading as a prototype. No single change there was big.
The sum of them is the difference between "a clever demo" and "a thing that looks
like someone meant it."

That's the quiet leverage of style work. A pile of changes that each touch zero
behavior can still change how the product *feels* by a lot, because most of
"feels finished" is just things being consistent and where you expect them. It's
the cheapest quality you can buy, and it's almost always undervalued because no
individual diff in it looks like it mattered.

That polish is what made the rest of the day — the part where it actually became
a release — feel like it was shipping something finished. [That's the other
half.](/blog/blue-steel-is-out)
