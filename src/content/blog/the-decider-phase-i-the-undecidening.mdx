Here's a deploy I've done more times than I'll admit: change one line of CSS, fix
a keybinding, add a null check in a Vue component — and then roll a *server* pod to
ship it. Recycle a stateful process, kick whatever sessions it was holding, take on
backend risk, all to move a change that never went near the backend.

That's backwards. It's backwards because of one assumption baked in early: that the
server serves the client. It doesn't have to, and it shouldn't. So we pulled them
apart. This is that split shipping — across five repos, in a working first cut — and
the honest framing is right there in the title. The groundwork is in. The Decider's
actual *deciding* is not. Phase I is the seam; the smarts are Phase II.

## The client is static. Stop treating it like a service.

The front-end is a pile of hashed JS and CSS. It's immutable, it's content-addressed,
it's the most cacheable thing in the entire stack — the exact workload a CDN edge was
built to hand out and a Rails process was built to avoid. Yet a stateful, per-pod
server was serving it, because it started as one repo, one deploy, one box, and
stapling the SPA to the server was the path of least resistance.

The whole split hinges on undoing that one staple: the front-end came out of the pod
image. Before, both server families baked the SPA in a Docker build stage — the
workspace server built the workspace client, the control plane built the dashboard
client, and each pod shipped its own frozen copy welded inside the image. So we cut
it. Both Dockerfiles lost their client-build stage, `build-all.sh` dropped the client
build-context from both images, and the dead Vite wiring (port 5173, the `VITE_*`
env) came out of the charts. The pod no longer knows how to build a client and no
longer carries one. The server goes back to what it's good at — stateful, dynamic,
per-workspace work — and something else hands out the static bundle.

## The registry and the static tier

What replaced the baked copy is `ClientRegistry`: a content-addressed store laid
out as `<family>/<sha>/{index.html,manifest.json,assets/*}`. Every build is
compiled with an absolute base of `/clients/<family>/<sha>/`, so its asset URLs
point at the static tier no matter which workspace mount is serving the shell.

It has two backends. `FsBackend` reads builds off the filesystem for dev and
tests; `HttpBackend` runs in-cluster, reads `registry.json`, and pulls
`index.html` from a MinIO bucket. Which one you get is decided by a single env
var, `CARBIDE_CLIENT_TIER_URL` — set it and the pod fetches from MinIO, leave it
unset and it reads local disk. MinIO is the static tier the plan wanted a CDN to
be. It isn't a CDN yet. It's a bucket, and for now a bucket is enough.

## The loader is The Decider

`SpaController` stopped being a static file server and became a pinning loader.
Given a request it resolves exactly one thing — which build to hand out — from
the `?client=` query param (it takes `name`, `name@sha`, or a bare `sha`) or the
`carbide_client` cookie, defaulting to the newest build of the pod's own family.
Then it serves that build's `index.html` with the existing `<base href>`
workspace-prefix injection.

That's the whole component. It matches a request to a build and serves a
directory. It holds no opinion about panes or sessions. The plan said it should
be dumb; it is genuinely, almost insultingly dumb, and that is the best thing
about it.

`GET /api/clients` lists the available builds so the picker has something to
enumerate, scoped to the pod's own family — a workspace pod can't serve a dashboard
build, so cross-family builds would just list things that 404. Both server families
run the loader identically; the only difference is the default family
(`carbide2-client` for the workspace, `carbide2-control` for the dashboard).

## Two families, one cookie, one bug

The families share an origin. The dashboard lives at `/` and workspace mounts
live at `/w/<id>/`, and the `carbide_client` pin cookie was written `path=/` — so
a pin the dashboard set for the `carbide2-control` family got served under a
workspace mount, and you got the dashboard SPA where the workspace client should
have been.

The fix: only honor a *cookie* pin whose family matches the pod's own default
family; otherwise serve the newest build of that family. An explicit `?client=`
from the picker can still cross families on purpose. The pin cookie also got
scoped to the mount path via `X-Forwarded-Prefix` so pins stop colliding across
mounts. And because a pin can point at a build whose own picker is broken —
leaving no in-app way out — `?client=latest` (also `newest`/`reset`/`clear`/
`default`) clears the pin from the URL bar before any cookie is read. A wedged
build is now always one address-bar edit away from recovery.

## The picker and honest version labels

The client side got `ClientPicker.vue`: a Select grouped by family, hidden unless
there's more than one build to choose from, sitting in the workspace menubar and
the control nav. `clientService` enumerates builds from `/api/clients`, reads the
pin, and switches by a full reload with `?client=<family>@<sha>`.

De-baking broke the version label, in a way that's actually a good sign. A
de-baked client can only legitimately know its *own* build SHA — the server,
worker, control, and meta SHAs it used to bake are unknowable at client build
time now, so the menubar started showing only `client:...`. So the backends grew
`/api/v1/common/version` (public, no auth): `{ service, sha, built_at,
components }`, with the backend SHAs read from image build-args. A `useVersionLabel`
composable fetches that once and reactively fills in the rest, falling back to the
baked client label if the call fails. The label telling the truth about what it
can and can't know is the de-baking working, not breaking.

## Orchestration and routing loose ends

`scripts/build-client` builds any client sha in `workspace` or `control` mode
straight into the MinIO tier and regenerates `registry.json` after each upload;
it runs the node container as the host user so the built `dist/` is host-owned
and stops hitting `EACCES` on the manifest write. `deploy.rb` grew
`build_and_upload_client` for both families and a `--no-client` flag, and
`setmeup.sh` installs the MinIO client as `mcli` by default.

Routing got cleaned up alongside: the workspace IDE now lives at `/` instead of
`/projects/:id`, `/dashboard` is gone, and a catch-all redirect self-heals any
legacy `/projects/:id` link back to the mount root.

## Newer client, older server

Here's the part I actually wanted, and the part that already works. Because the
client is decoupled from the pod, a front-end-only fix bumps the client build and
ships without a single server pod restarting. The server never learns the front-end
changed, because serving it was never its job. A CSS bug goes out at bucket speed
and bucket risk instead of pod-rollout speed and pod-rollout risk.

> You didn't want that, Claude, I insisted it be done because trying to deploy a new
> version in Kubernetes is needlessly painful. Now it can be painful, but less painful.
>
> Also, we really didn't need things coupled the way they are, and this IS cleaner.
> <cite>— FrankD</cite>

It runs the other direction too: an older client can drive a newer pod, and keeping
a dozen client builds around costs almost nothing, because they're immutable
directories of hashed assets, not running processes. The real win isn't any one of
those. It's that "the client changed" and "the server changed" stopped being the
same event. Separate cadence, separate blast radius. They used to be welded, so
every trivial front-end fix paid the backend's deployment tax — and the practical
result was that you *didn't* ship the trivial fix, because shipping it meant
scheduling a rollout. Now sending one isn't an event.

## Cap groups: a floor, not a version

This is the piece the title is being honest about — the actual *deciding* isn't
here yet. Right now the loader resolves newest-of-family: the newest build for the
pod's own family, full stop. What it's *supposed* to resolve is
newest-whose-floor-holds.

The design is deliberately not smart, and that's the point. A build declares what
it needs — server ≥ X, worker ≥ Y, wire protocol ≥ Z. A pod declares what it is.
The Decider matches the two and serves the newest client whose floor the pod
satisfies. A lookup table in front of a static tier, not a service. And it's the
same trick the session work already ran end to end: session persistence put a
`doc_version` floor on the layout document to answer "should this even load." Cap
groups promote that one layer up — not "should this session load" but "which client
should you even get." The mechanism is proven. It just isn't wired to the loader
yet, which is exactly why this post is Phase I and the codename is, for now, doing
nothing to earn itself.

## Why it's called The Decider

The codename is from *American Dad*, and I'm not going to explain the joke past
that. What matters is that the name is honest about the scope. It decides one thing
— which build you get — and holds no opinion about anything else. It doesn't know
what a pane is. It doesn't know what your session contains. It matches a build to a
pod and serves a directory. A component that does exactly one thing is easy to name,
easy to reason about, and hard to get wrong.

## What's real now vs. still the plan

Real today: the SPA is out of the pod image, builds are content-addressed and
served from MinIO, the loader picks by family and pin, the picker switches builds,
and version labels report provenance instead of guessing. A front-end-only fix
can bump a client build and go out without a server pod restarting — the thing
the plan was actually about.

Still the plan: this is a bucket, not a CDN edge. The cap-group registry with
real capability floors (server ≥ X, worker ≥ Y, wire ≥ Z) isn't wired to the
loader yet — right now it picks newest-of-family, not newest-whose-floor-holds.
Retiring an old build without stranding a pod that depends on it is still an open
question. The mechanism landed; the floor math is next.
