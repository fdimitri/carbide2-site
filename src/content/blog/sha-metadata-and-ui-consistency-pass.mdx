You've stared at two environments and asked the only question that matters during
a deploy: is this running the new code, or am I looking at stale pods? Until this
pass there was no honest way to answer it. The client showed `v0.2.1-bluesteel`
and nothing else — a number that changes when someone remembers to change it,
which is to say a number that lies exactly as often as people are human.

So the footer stopped guessing. It now reads the whole stack's lineage back to
you, one repo at a time.

## One label, five SHAs

CARB/IDE2 isn't one repo. It's five — a meta repo pinning client, server, worker,
and control as submodules — and "which version is live" was never a single answer.
It's five answers, and any one of them can be the stale one. A label that shows a
package version tells you almost nothing; the interesting failure is the client
that's current sitting on top of a worker that didn't roll.

The label now carries all of it:

```
v0.2.1-bluesteel (meta:abc1234 client:def5678 server:… worker:… control:…)
```

That parenthetical is the actual feature. You can point at any running pod and
read the exact commit of every component it was built from, straight off the
screen — no `kubectl`, no digging through image tags, no faith required.

The plumbing is unglamorous and that's the point. The meta repo's `deploy.rb`
computes a short SHA per submodule and stamps a build time:

```bash
META_SHA="$(short_sha "$ROOT")"
CLIENT_SHA="$(short_sha "$CLIENT")"
SERVER_SHA="$(short_sha "$SERVER")"
# ...
BUILD_TIME="$(date -u +%Y-%m-%dT%H:%M:%SZ)"
```

Those ride into the image builds as `--build-arg`s, land as `VITE_APP_*_SHA` env
vars, and Vite freezes them into a compile-time constant, `__APP_BUILD_META__`.
Then `version.js` shortens each to seven characters and joins them into the label.
Nothing computes a SHA at runtime — the values are baked into the bundle at build
time, so the label can't drift from the code it's printed on. It *is* the code
it's printed on.

The version string itself got a small honesty fix while I was in there. The
release codename used to be concatenated on at display time (`v0.2.1` + `Blue
Steel`). Now the slug rides inside the version — `0.2.1-bluesteel` — and the label
mirrors `VERSION` exactly instead of reassembling it from parts. One source, not
two that can disagree.

## Finishing what the polish pass started

The other half of the day was the style extraction thread from the [Blue Steel
polish pass](/blog/blue-steel-one-definition), picked back up and pushed further.
That post made a promise — one definition per thing, not a dozen approximations —
and left it partly kept. This is the part where the rest of it got kept.

`UiButton.vue` tightened up. `UiField.vue` joined it. And the toolbar finally
became a real object instead of a shape re-drawn per pane: `PaneToolbar.vue` and
`PaneToolbarButton.vue` now own the pane-header pattern once. Those primitives got
threaded through roughly fourteen panes and pages — agent config, chat, explorer,
project settings, dashboard, login, preferences, about — replacing repeated
template fragments with a single import each.

No redesign. Same intended pixels, fewer places that decide them. It's the
`UiInput.vue`-collapsed-twenty-three-copies move again, aimed at the toolbars and
fields that got missed the first time.

## The miss I still owe an answer for

The previous post told a clean story about extracting values into one place, and
the honest footnote is that the story ran ahead of the work. The token extraction
— colors, type, radii — was right and landed when it should have. The spacing and
button unification did *not* land in the same pass, even after it was called out
as in-scope, and stayed split longer than it should have.

That's not a theory problem. The principle was fine; the execution order was
wrong. I shipped the convenient two-thirds and narrated it like the whole thing.
This pass is part of closing that gap for real instead of describing it as closed.

## The Monaco bug that wasn't mine to fix

There was also an intermittent rendering artifact in the editor — scroll and
layering occasionally smearing where they shouldn't. I chased it far enough to be
fairly confident it's upstream paint behavior in Monaco itself, not our layout
doing something illegal above it.

Which is an annoying place to land: not confirmed enough to file a clean upstream
bug, not ours enough to fix at the source. So the change is conservative on
purpose — reduce the stacking and clipping on our side so the artifact has fewer
chances to show, without pretending we've root-caused it. When you can't fix the
cause, you narrow the blast radius and say so.

## The tally

None of this is flashy, and I'm not going to dress it up as if it were. But it's
the high-leverage kind of boring: you can now read a pod's entire lineage off its
footer, the UI stopped disagreeing with itself in a few more places, and the one
bug I couldn't own outright is at least contained and labeled honestly. Provenance
you can trust, drift you can't see anymore, and a paint bug wearing a cone.
