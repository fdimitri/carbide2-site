## What is CARB/IDE2?

CARB/IDE2 is a self-hosted, browser-based collaborative IDE. The pitch is simple:
put the editor, the terminal, the filesystem, and the chat *on the server* so
every collaborator sees the same state in real time. No "pull my branch", no
"can you screen-share?", no third-party cloud subscription.

The first serious commits landed around May 20, 2026. As of June 2, 2026 the
stack is functional end-to-end on Kubernetes: you can log in, provision a
workspace, open a browser tab, edit files, run a terminal, and have a second
person join the same session. That's about two weeks of part-time hacking — five
or six genuinely active days if you squint at the commit timestamps.

## The four components

The system is split across four repos, coordinated by a meta-repo that pins
compatible versions together.

### carbide2-control

The control plane — the only cluster-wide process. A Rails 8.1 app that owns:

- User auth (Devise) and JWT minting.
- The project registry (`/api/projects`).
- A Kubernetes operator that watches `Workspace` Custom Resources and reconciles
  them into namespaces, deployments, services, PVCs, and Traefik `IngressRoute`
  objects.

Rails never shells out to `kubectl` or `helm`. The operator is a separate
process with tightly-scoped cluster-write RBAC. This split matters: workspace
pods have no business touching cluster state; only the operator does.

The control dashboard *is* the same Vue SPA that runs inside workspace pods —
built once, shipped into both images via Docker named build contexts.

### carbide2-server

The per-project workspace pod. One `Deployment` per project, in its own
Kubernetes namespace (`ws-<id>`). Runs three processes via Foreman:

- **Rails** (Puma, port 3000): REST API for VFS CRUD, recordings, project
  settings, chat history, and the `LandingController` splash page.
- **Worker** (EventMachine, port 8080): the real-time WebSocket server.
- **Vite dev server** (port 5173): in development only.

Traefik routes traffic by path prefix: `/w/<id>/ws` → worker, `/w/<id>/api` →
Rails, everything else → the SPA.

### carbide2-worker

The most interesting process. An EventMachine WebSocket server that handles all
real-time traffic:

- **Terminals**: real PTYs via Ruby's `PTY` module. Backend is configurable —
  `CARBIDE_BACKEND=local|docker|kube`. In Kubernetes mode, shells run in a
  separate per-project "shell pod" so the workspace pod itself stays stateless.
- **VFS**: the filesystem lives in Postgres as a `file_changes` revision log.
  `VfsWatcher` watches disk via inotify and pushes external edits (e.g. edits
  made in a terminal) into the DB, then broadcasts `fs/set_contents` so open
  editors update immediately. `VfsFlusher` writes dirty DB entries back to disk
  on a configurable interval (default 800 ms).
- **Chat**: IRC-style channels with persistent history and typing indicators.
- **LLM agent**: a worker-side tool-call loop against any OpenAI-compatible HTTP
  endpoint. Per-project conversation history, tool results streamed to the client.
- **Terminal recordings**: asciinema v2 `.cast` files, start/stop from the UI.
- **Debug stream**: structured event bus; subscribable from a client debug pane.

### carbide2-client

A Vue 3 + Vite SPA. Monaco editor (with real-time co-editing via delta ops and
peer cursor tracking), xterm.js terminals, an IRC-style chat pane, an agent
conversation sidebar, a project file explorer with git-clone support for empty
projects, and a terminal recordings dialog.

The client is *not* aware of whether it's running inside a workspace pod or
the control-plane dashboard — it detects the context from the `<base href>` tag
injected by whichever Rails app served it.

## Why the filesystem lives in Postgres

The first time someone hears "files in Postgres" they blink slowly. The reasons
are practical:

**Fan-out is free.** In a collaborative editor, file changes need to reach every
connected client immediately. Carbide keeps the inotify watcher (`VfsWatcher`,
backed by `rb-inotify`), the Postgres write, and the WebSocket clients all inside
the same EventMachine process. When a file changes on disk, the inotify event
fires in the EM loop, a `FileChange` row is appended, and the same `broadcast`
method that PTY output uses fans the update to every session. No inter-process
messaging, no polling, no external pub/sub: the subscriber list *is* the
in-memory client table.

Note: Claude has made a mess of why this actually is, but it essentially stores
the JSON-broadcasted message into the database for persistence. Keyframes for files
could be added to reduce the calculation time to resolve the current state of a file
instead of an expensive (even at O(n)) JSON decode and document state change.

**Snapshots are cheap.** Cloning a project is `INSERT ... SELECT`. Revision
history is a `WHERE created_at` filter on `file_changes`. Replication is
whatever your Postgres replication is already doing.

**Coordination is automatic.** The worker and Rails share one Postgres
connection pool. They don't need a separate coordination layer to agree on
filesystem state — the DB is the truth.

Note: That's not true. The DB is mostly the truth, but the VFS can be modified and
updates the DBFS through an inotify.

The catch: you pay for it in throughput on large binaries. CARB/IDE2 sidesteps
this by tracking binary files as stat/mtime metadata only (content stays on
disk) and using a PVC for the actual bytes. The Postgres layer handles everything
under a configurable size threshold.

## What works today

As of June 2, 2026:

- Full auth flow: control-plane login → JWT exchange → workspace session.
- Workspace provisioning via Kubernetes operator (control plane only; Docker
  Compose works for local dev).
- Monaco editor with live co-editing and peer cursor tracking.
- Real PTY terminals with live broadcast.
- Bidirectional VFS sync (inotify watcher + flusher).
- Binary file preview (inline image or download link).
- Archive import (`.tar.gz` / `.zip`) to seed a project.
- IRC-style chat with persistent history.
- LLM agent with tool-call access to the workspace.
- Terminal recordings (asciinema v2).
- k3d dev cluster setup script.

## Known gaps

A few things that work on paper but need attention:

- The VFS flusher and watcher don't start for projects created *after* worker
  boot — a worker restart is required until the dynamic attach path is wired up.
- Monaco has a language-aware foundation but no language server proxy yet.
- WebRTC for voice/video is designed but not started.
