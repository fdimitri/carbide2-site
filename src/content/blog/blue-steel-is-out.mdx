[`v0.2.0-bluesteel`](https://github.com/fdimitri/carbide2/releases/tag/v0.2.0-bluesteel)
is tagged. Same version string across all five repos, the meta repo pinning the
submodules at the validated SHAs, a GitHub release with the component table
written out. It's a real, reproducible point in history — and the stack it pins
was deployed end-to-end on a box that had never seen the code.

The code lives at [github.com/fdimitri/carbide2](https://github.com/fdimitri/carbide2);
[`INSTALL.md`](https://github.com/fdimitri/carbide2/blob/main/INSTALL.md) takes a
clean Linux host from nothing to a running stack (tested on Ubuntu 24.04).

The [polish pass](/blog/blue-steel-one-definition) was the visible
half. This is the half that makes it a release instead of a Tuesday: the deploy
ergonomics and the cut itself.

## A failure that couldn't be proved

The day's first problem had no evidence left to examine. Terminals in a running
workspace had stopped working overnight, and by the time anyone looked the pod
that failed was gone — reaped hours earlier, logs gone with it, and a redeploy in
between had overwritten anything that might have survived. The honest answer was
**unprovable**.

So rather than invent a root cause, I fixed the thing that had actually failed:
there was no log to read. Workers now write to a persistent log on the project
volume that outlives the pod, with a periodic heartbeat and a recorded reason for
exit. This didn't solve the terminal failure — it makes the next one solvable. On
a release day that's the more useful of the two.

## Teaching the deployer to stop shouting

Standing the stack up on a fresh machine, the first thing it did was scream. Not
because anything was wrong: `deploy.rb` was piping every line of `docker build`,
every `k3d image import`, and a multi-screen JSON dump straight to the console. A
lot of that lands on stderr, which the pretty-printer renders in an alarming red.
So a *successful* cold-cache deploy — which, on a fresh box, builds Ruby from
source and takes its time — looked exactly like a stack trace.

Long, noisy external commands now run through a silent printer behind a single
friendly line:

```
==> building the three container images — on a cold cache this builds Ruby
    from source, so give it a few minutes (reticulating splines...)
```

The output is captured, not discarded — if a build actually fails, the full log
prints right where you need it. But on success you get a calm sequence of progress
lines instead of a wall of red you have to *not* panic about.

> The default emotional register of a build log should not be "something has
> gone horribly wrong." It should be "this is working; here's where we are."

## What shipped

The release covers the workspace client — the Vue SPA — plus the backend
components it talks to:

- **Collaborative Monaco editor**, wired to the live in-database filesystem;
  edits persist continuously, no separate save.
- **Shared terminal** — PTY output streamed into the pane.
- **Agent pane** — chat with an in-workspace AI agent; tool calls render as
  collapsible rows grouped under the agent turn.
- **Chat pane** — per-project channels with presence and avatars.
- **Agent config** — finally a UI for it, instead of a `rails console`.

This is a `0.x` cut. Expect rough edges and breaking changes between releases.

## What's next

`v0.2.0-bluesteel` is the release — that's the line in the sand. From here it's
patch releases as needed, and back to the actual backlog: more features and a
healthy pile of [open issues](https://github.com/fdimitri/carbide2/issues) to
work through.
