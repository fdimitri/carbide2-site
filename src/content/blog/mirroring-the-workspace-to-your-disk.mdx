You've got a workspace. It lives in a pod somewhere, behind a network you don't
control. And you want to run `npm run dev` against it. Or a Rails server. Or
ripgrep, or your own editor, or any of the tools already sitting on your laptop
that have never heard of CARB/IDE and never will.

How?

The first answer is a proxy. Expose the port the dev server binds, route it back
out through an HTTPS ingress, and now your local browser can reach a process
running in the pod. That works. It also means solving network ingress for every
port a user might decide to bind, on demand, per workspace — which is a whole
project with its own security surface and its own failure modes. You're building
a tunnel so the compute can stay put.

So flip it. Don't drag the compute out to your laptop. Bring the files in. Run
the dev server locally, against a local copy of the workspace, and keep that copy
current. No ingress, no tunnel, no per-port anything. Your tools run where your
tools already live.

That copy needs a keeper. It's a new client: a read-only mirror daemon.

## What it does

You launch it from a shell. It reads a config file — which workspace, which
project, which local directory to fill — and then it goes quiet and does two
things forever.

First, a full sync. It walks the project's file tree and writes every document
to disk under your mirror root, pruning anything local that no longer exists
upstream. Now you have the whole workspace as plain files.

Then it subscribes. It opens a websocket, sends an `open` for every file it just
wrote, and follows the revision stream. Text edits arrive as deltas and get
applied to the file in place. Full-content replacements, creates, renames,
deletes, binary updates — each one maps to the obvious filesystem operation,
written atomically through a temp file so a reader never catches a half-written
document. Somebody edits a file in the browser; a beat later it changes on your
disk. That's the entire product.

## One-way, on purpose

The mirror only flows one way: workspace to disk. It will never take your local
edits and push them back up. That's not a missing feature I'll get to later. It's
the design.

Two-way sync sounds like the generous version, but it's the same problem we
already have one layer down, between the on-disk VFS and the database-backed
filesystem, scaled up to the whole workspace. The moment both ends can write, you
need an authority: when your local `README.md` and the workspace `README.md`
disagree, which one is right, and as of when? A mirror has no basis to answer
that. It wasn't there for the edits. It has a copy and a stream, not a history it
can arbitrate.

So it doesn't try. It's a read-only replica, and it says so. You get a live local
copy to build against. You do not get a sync engine that guesses. If you want to
change a file, change it in the workspace — the one place that actually owns the
document.

## Then it didn't work (three times)

The shape came together fast. Getting it to run took three separate faceplants,
and they're the good part.

**The five-minute token.** The first version wanted you to paste an auth token
into the config. The token the workspace hands out for this lives for five
minutes. So you'd configure the daemon, start it, go get coffee, and come back to
a dead process holding an expired credential. Absurd for something whose entire
job is to run unattended for days.

The fix was to stop treating auth as a value you paste and start treating it as
something the daemon does. It logs in to the control plane with real credentials,
gets a durable token, and exchanges that at the workspace's own login for a
short-lived working token — then refreshes it before it expires and retries once
on any 401. The five-minute lifetime is still there. It's just the daemon's
problem now, not yours, and you never see it.

**Zero files.** First clean run: `full sync complete: 0 files`. The project was
not empty. I was staring at two files and a directory that very much existed.

The tree endpoint returns a single root node with the path `/`, and every real
file hangs off its `children`. The walk had a guard to skip the root — reasonable,
you don't want to write a file called `/` — but the guard fired *before* the
recursion, so it skipped the root and everything under it in one move. It walked
to the door, decided the door wasn't a file, and went home. Let the root fall
through to its children and the whole tree shows up.

**The websocket that talked to the wrong object.** With files finally syncing,
opening the stream crashed instantly: `undefined method 'info' for nil`, from a
line where `@logger` is obviously not nil. Except it was. The websocket library
runs your event callbacks with `instance_exec` against the socket object, so
inside `on(:open)` the `self` isn't the daemon — it's the client. Every instance
variable I reached for belonged to the wrong object and came back empty.

The fix is to stop depending on `self` in those blocks. Hand the callbacks bound
method objects that carry their own receiver, and they run as the daemon no
matter what `self` the library swaps in. Same trick pays off twice: those bound
methods also update the connection-state flags the CLI reads.

## Asking the daemon what it knows

Once the thing runs as a background daemon, you can't see it anymore. Is it
connected? Is this file actually on the stream, or just a stale copy from the last
full sync that's been quietly frozen for an hour? The files on disk can't tell
you. Only the daemon knows, and it wasn't saying.

So it says now. The daemon listens on a Unix socket, and there's a small `carbide`
CLI that asks it questions:

```
$ carbide fs-get info ./README.md
README.md
  workspace path:   /README.md
  local path:       /home/frankd/ws1-test/README.md
  subscribed:       yes
  connected:        yes
  revision:         1519
  server updated:   2026-07-21T22:21:03.498Z
```

`connected` and `subscribed` come from the daemon's own live state — it's the only
process that knows whether the socket is up and which paths it's actually
following. The revision and timestamp it fetches from the workspace on demand.
The CLI just translates your shell path into a workspace path and asks; the daemon
answers because it's the only one who can.

## The part that isn't done

Be clear about what this buys and what it doesn't. It sidesteps ingress; it does
not solve it. Running your dev server locally against mirrored files is great
until the thing you actually need is the port itself reachable from somewhere
else — a webhook, a phone on a different network, a teammate. For that you still
want the real answer: stable VIPs and a mesh VPN into the workspace, so a port in
the pod is just an address you can hit. That's still on the list. The mirror
buys you the local dev loop today without waiting for the network to grow up.

And it's read-only, still on purpose. The day there's a principled answer for who
wins a conflict, write-back can exist. Until then a mirror that guesses is worse
than a mirror that's honest about being a copy.
